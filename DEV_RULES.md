# PetSpace Development Rules

이 문서는 **PetSpace 개발의 유일한 기준 문서**이다.  
Claude, Codex CLI, 사람 작업자는 모두 이 문서를 최우선 기준으로 따른다.

- 별도의 `CLAUDE.md`, `AGENTS.md`, 기타 규칙 문서는 사용하지 않는다.
- 기존 `Cloade.md` 내용은 이 문서로 통합한다.
- 작업 전/중/후 모든 판단은 이 문서 기준으로 한다.

---

## 1. 목표

PetSpace는 WordPress + Elementor + ACF + 커스텀 코어 플러그인(`pet-space-core`) 기반으로 개발한다.  
이 문서의 목적은 아래와 같다.

1. 기능이 여러 곳에 흩어져 다시 꼬이지 않게 한다.
2. 구조, 권한, 데이터, UI, 보안, 스타일 규칙을 하나로 통일한다.
3. 새 기능을 추가할 때 기존 기능을 깨지 않도록 한다.
4. Claude와 Codex가 항상 같은 기준으로 일하게 만든다.

---

## 2. 최상위 원칙

### 2.1 코드는 한 곳에만 둔다
- 확정 기능은 반드시 `pet-space-core` 플러그인 안에서만 관리한다.
- `functions.php`, Code Snippets, 임시 삽입 코드는 테스트용으로만 사용한다.
- 테스트가 끝난 코드는 반드시 코어 플러그인으로 이관한다.

### 2.2 기능 하나는 책임 소스 하나만 가진다
- 동일한 기능을 여러 파일, 여러 계층, 여러 방식으로 중복 구현하지 않는다.
- 예:
  - `[store_region]` 숏코드는 한 곳에서만 정의
  - `[store_hours]` 숏코드는 한 곳에서만 정의
  - premium-only-group 토글 JS는 한 곳에서만 정의

### 2.3 한 번에 하나의 기능만 끝낸다
항상 아래 순서로 진행한다.

1. 정책/룰 확정
2. 데이터 구조 확정
3. Elementor 슬롯/화면 자리 확보
4. 코어 모듈 구현
5. 테스트
6. 문서화

### 2.4 새 기능은 신규 규칙을 따른다
- 기존 레거시 파일은 당장 전부 뜯어고치지 않는다.
- 단, **새 기능은 반드시 이 문서의 신규 규칙**으로만 추가한다.
- 기존 기능을 크게 수정할 때만 새 구조로 옮긴다.

## 3. 프로젝트 핵심 도메인 규칙

### 3.1 Store 중심 구조
- 매장 CPT slug: `store`
- 매장 taxonomy: `store_cat`
- 입점사 소유권 판단:
  - `store.post_author == vendor user_id`

### 3.2 플랜은 계정이 아니라 매장 단위
- 무료/프리미엄은 계정 role이 아니라 **매장 메타(`store_plan`)** 로 관리한다.
- 이유:
  - 1계정이 여러 매장을 가질 수 있기 때문
- 따라서 모든 프리미엄 기능은 반드시 현재 store 기준으로 판정한다.

### 3.3 역할(role)의 의미
- `vendor` role은 “입점사 여부”를 나타낸다.
- 유료/무료 여부는 role이 아니라 `store_plan`으로 판단한다.
- role은 접근 가능 범위를 나누는 보조 레이어로만 사용한다.

### 3.4 다매장 지원은 기본값
- 1개의 vendor 계정이 여러 개의 store를 소유할 수 있다.
- 모든 대시보드/관리 기능은 “현재 선택한 store”를 기준으로 동작해야 한다.

## 4. 코어 플러그인 구조 규칙

### 4.1 코어 플러그인 고정
- 모든 영속 기능은 `pet-space-core` 안에서만 구현한다.
- 플러그인 루트 예시:

```text
pet-space-core/
├─ pet-space-core.php
├─ includes/
├─ modules/
└─ assets/
4.2 신규 모듈은 폴더형 구조만 허용

새 모듈은 반드시 아래 구조를 따른다.

modules/{name}/
├─ {name}.php
├─ assets.php
└─ assets/
   ├─ css/
   │  └─ {name}.css
   └─ js/
      └─ {name}.js

예:

modules/favorites/
├─ favorites.php
├─ assets.php
└─ assets/
   ├─ css/favorites.css
   └─ js/favorites.js
4.3 레거시 파일 규칙

현재 이미 존재하는 평면 파일형 모듈(예: notice.php, coupon.php, vendor-dashboard.php)은 레거시로 취급한다.

즉시 이동하지 않는다.

버그 수정은 허용한다.

구조 개편은 해당 기능을 크게 손볼 때만 진행한다.

새 기능은 절대 평면 파일 방식으로 추가하지 않는다.

4.4 로딩 방식

파일 로드는 psc_require() 같은 공통 로더를 통해서만 진행한다.

신규 폴더형 모듈은

assets.php

{name}.php
순으로 로드한다.

누락 파일이 있어도 전체 사이트가 fatal error로 죽지 않게 방어한다.

5. ACF 규칙
5.1 ACF 필드명은 계약(contract)이다

field name은 한 번 확정하면 쉽게 바꾸지 않는다.

새 기능 개발 전 반드시 ACF 필드명 목록을 먼저 문서화한다.

기능 구현은 필드명이 확정된 뒤 시작한다.

5.2 현재 핵심 ACF 필드
store 관련

store_address

store_phone

store_plan

store_gallery_basic

store_gallery_premium

store_hours_weekly

store_temp_closures

운영시간 구조

store_hours_weekly

mon, tue, wed, thu, fri, sat, sun

각 요일 하위:

closed

open

close

break_start

break_end

note

5.3 프리미엄 전용 필드

프리미엄 필드는 별도 그룹으로 분리한다.

관리자 화면에서는 store_plan 값에 따라 .premium-only-group 을 숨김/표시한다.

프리미엄 만료 시 데이터는 삭제하지 않고 유지한다.

단지 UI/기능만 잠근다.

6. Elementor 규칙

6.2 로직은 Elementor에 넣지 않는다

아래는 Elementor가 아니라 코어 모듈이 담당한다.

6.3 데이터 제공 방식

7. UI / 스타일 규칙
7.1 인라인 CSS 금지

신규 코드에서는 인라인 CSS를 금지한다.

스타일은 반드시 외부 CSS 파일로 분리한다.

7.2 인라인 JS 최소화

신규 코드의 JS도 가능하면 외부 JS 파일로 분리한다.

다만, WordPress admin에서 field key 같은 동적 값 주입이 꼭 필요할 때만 wp_add_inline_script() 허용

7.3 셀렉터 규칙

태그 단독 셀렉터 사용 금지

반드시 .psc-* 접두 클래스 또는 모듈 네임스페이스 클래스와 결합

예:

.psc-store-hours

.psc-favorite-button

7.4 !important 사용 규칙

글로벌 테마/엘리멘터와 충돌하는 버튼/폼/토글 핵심 속성에 한해 제한적으로 허용

남발 금지

7.5 hover 시 color 명시 규칙 — 테마가 button:hover { color: #fff } 같은 전역 스타일을 가지므로, hover 규칙에도 반드시 color !important를 명시한다.

8. 디자인 참고 원칙 (Toss UI 포함)
8.1 기본 원칙

PetSpace는 Toss UI를 포함한 우수 서비스의 사용성 원칙과 패턴은 참고할 수 있다.

단, 특정 서비스의 표현(expression) 을 그대로 복제하지 않는다.

참고 대상은 “아이디어/패턴”이지, “화면 결과물의 동일·유사 복제”가 아니다.

8.2 참고 가능한 요소

다음 요소는 일반적인 UX 패턴으로 참고 가능하다.

정보 우선순위가 분명한 카드형 구성

넓은 여백과 단순한 정보 위계

직관적인 CTA 버튼 배치

짧고 명확한 마이크로카피

최소 단계 플로우

요약 → 상세 확장 구조

리스트/카드/모달/바텀시트/오프캔버스 같은 상호작용 패턴

폼 입력 UX, 검증 UX, 상태 뱃지 UX

8.3 금지 사항

다음은 금지한다.

Toss 화면을 스크린샷 기준으로 그대로 재현하는 것

색상, 폰트 체계, 카드 모양, 아이콘 스타일, 여백, 버튼, 문구, 애니메이션을 조합하여
전체 인상이 Toss와 유사하게 보이도록 만드는 것

Toss의 고유 카피/문구/용어를 그대로 사용하는 것

Toss의 고유 아이콘/일러스트/이미지/에셋을 복제하거나 유사하게 재현하는 것

특정 핵심 화면을 레이아웃 수준에서 거의 동일하게 만드는 것

사용자가 PetSpace를 Toss와 제휴/연동된 서비스로 오해하게 만드는 표현

8.4 PetSpace 디자인 방향

PetSpace는 아래 방향으로 독립적인 디자인 언어를 만든다.

브랜드 컬러, 포인트 컬러, 상태 컬러를 Toss와 다르게 설정

카드 라운드, 그림자, 간격, 버튼 톤을 자체 기준으로 통일

반려동물 플랫폼 특성에 맞는 친근하고 따뜻한 톤 유지

로컬 서비스/매장 탐색에 맞는 UI 밀도와 카피를 사용

“매장/반려견/쿠폰/공지/접종지갑” 같은 PetSpace 고유 도메인 구조를 전면화

8.5 실무 체크리스트

디자인/프론트 작업 시 아래를 확인한다.

이 화면을 봤을 때 일반 사용자가 “토스랑 거의 같다”고 느낄 정도인가?

색상/카드/버튼/간격/카피까지 합쳐서 전체 인상이 유사한가?

해당 화면의 핵심 레이아웃이 특정 Toss 화면과 1:1로 대응되는가?

PetSpace 고유 브랜드 요소 없이도 성립하는 화면인가?

하나라도 “예”에 가까우면 재설계한다.

8.6 구현 지침

레퍼런스 수집 시 “왜 좋은지(패턴/원칙)”만 기록하고, 실제 화면은 그대로 옮기지 않는다.

구현 전 “참고 포인트”와 “차별화 포인트”를 함께 적는다.

컴포넌트 제작 시 PetSpace 고유 토큰(색상, radius, spacing, text scale)을 먼저 정의하고 사용한다.

신규 주요 화면은 최소 1회 이상 Toss 유사성 검토를 한다.

9.2 소유권 검사 원칙

vendor는 자기 소유 store 범위만 접근 가능

store 관련 파생 데이터(공지, 쿠폰, 리뷰 관리, 대시보드 등)는 반드시 해당 store 소유권과 연결해서 검사한다.

9.3 SQL 규칙

직접 쿼리 사용 시 반드시 $wpdb->prepare() 사용

prepare 없는 raw SQL 금지

9.4 REST / AJAX 규칙

privileged endpoint는 로그인/권한 없이 열지 않는다.

nopriv는 정말 공개로 열어야 하는 엔드포인트만 허용

9.5 출력 보안

속성: esc_attr

URL: esc_url

제한 HTML: wp_kses_post

9.6 업로드 / 첨부 규칙

업로드 파일은 무제한 공개 링크를 기본값으로 두지 않는다.

## 17. Claude + Codex CLI 협업 플로우

### 17.1 기본 원칙
- 기능 구현(Codex)과 디자인 명세(Claude)는 분리된 단계로 진행한다.
- CSS는 실제 구현된 코드를 확인한 뒤 작성한다.

### 17.2 작업 순서

**1단계: Codex 프롬프트 작성 (Claude)**
- 기능 요구사항, 파일 구조, PHP 로직, JS 동작, HTML 구조를 설명으로 전달
- 이 단계에서 CSS는 포함하지 않는다

**2단계: 기능 구현 (Codex)**
- 프롬프트에 따라 코드 구현 → 커밋

**3단계: 구조 파악 (Claude)**
- 커밋된 코드를 읽고 실제 HTML 구조, 클래스명, 마크업 계층을 확인

**4단계: CSS 디자인 명세 작성 (Claude)**
- 3단계에서 확인한 실제 구조에 맞춰 CSS 코드를 직접 작성
- Codex 프롬프트로 CSS 파일 생성 또는 직접 제공

### 17.3 이유
- Codex 구현 과정에서 클래스명, HTML 구조가 프롬프트와 달라질 수 있다
- CSS를 먼저 작성하면 셀렉터 불일치로 재작업이 발생한다
- 2-pass 방식이 1-pass 동시 전달보다 정확도가 높다

수정사항이 있는 경우 CODEX CLI에게 보낼 프롬으로 작성해서 보내줄 것
